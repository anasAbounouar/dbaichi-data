The Challenge: Manual Infrastructure Setup

Prior to this project, the GraalVM RISQ team managed OCI resources through manual processes, leading to significant challenges:

Time-Consuming Provisioning

Environment provisioning took hours per stack, slowing down test scenarios and team onboarding.

Human Error Risk

Manual configuration introduced errors and inconsistencies between environments.

No Version Control

Without version control, tracking changes or identifying who made them was difficult.

Configuration Drift

Inconsistent environments caused unexpected CI failures and "works on my machine" issues.

Poor Scalability

The manual process could not scale with increasing complexity of GraalVM's CI workflows.

Knowledge Silos

Infrastructure expertise was concentrated among a few team members, creating bottlenecks.

Four-Component Architecture

ConfigLoader

  • Loads stack-specific JSON configuration files
  • Performs initial validation and error handling
  • Provides unified interface for configuration access

ResourceFactory

  • Maintains registry mapping JSON sections to resource classes
  • Enables resource creation without hardcoded dependencies
  • Supports automatic registration of new resource types

DependencyResolver

  • Converts logical resource names to OCIDs
  • Handles both internal and external resource references
  • Ensures correct creation order of resources

ExecutionEngine

  • Orchestrates single-loop resource creation
  • Coordinates interactions between components
  • Generates pipeline outputs and exports

Design Patterns - Factory & Dependency Resolution

Factory Pattern

BaseResource abstract class defines a common interface for all OCI resource types
Concrete classes (VCNResource, SubnetResource, InstanceResource, etc.) inherit from BaseResource
Dynamic registration mechanism (register_all_resources()) creates a mapping between JSON configuration sections and Python implementation classes
Extensible without modifying core code - new resource types can be added seamlessly

The Factory Pattern enables dynamic resource creation without hardcoded dependencies on specific resource types.

Dependency Resolution

Logical names → OCIDs The DependencyResolver converts logical resource names (e.g., "vcn-prod") to actual Oracle Cloud Identifiers (OCIDs) at runtime
Automatic dependency tracking ensures resources are created in the correct order
Enforced creation order (VCN → Subnet → Instance)
Handles external OCIDs - OCIDs beginning with "ocid1." are passed through directly

Stack-Based Configuration

The framework uses a structured JSON format for defining infrastructure, enabling stack-based configuration management across different environments.

Configuration File Structure

$ find configs/ -type f
configs/
├── dev.json
├── staging.json
└── prod.json

JSON Configuration Example 

// configs/prod.json
{
  "metadata": {
    "name": "production-environment",
    "version": "1.0"
  },
  "vcns": [...],
  "subnets": [...],
  "instances": [...]
}

JSON Configuration Sections

  • metadata: Contains general information like name, version, and description of the stack.
  • vcns: Definitions for Virtual Cloud Networks.
  • subnets: Subnet configurations, often with references to VCNs.
  • instances: Compute instance specifications, including references to subnets.
  • security_lists: Firewall rules for network security.

Reference Resolution 

"vcn_id": "ref:vcn-prod"

The DependencyResolver converts this logical reference to the actual OCID of the "vcn-prod" VCN.

Sensitive Data 

# .env.local
OCI_CONFIG_FILE=~/.oci/config
OCI_PROFILE=DEFAULT

Sensitive data is managed through environment variables for secure credential handling.

Stateless Resource Design

Implementation adheres to architectural principles that promote maintainability and efficiency.

Architecture Principles

Stateless Classes

All methods are static (@staticmethod)

No Instance State

Resources don't store state, simplifying execution model

Pulumi Manages State

Infrastructure state in remote backend (OCI Object Storage)

Pure Functions

create() methods with inputs → outputs

Code Organization

$ tree src/
src/
├── __main__.py # Entry point, execution loop
├── config_loader.py # JSON parsing and validation
├── resource_factory.py # Factory pattern implementation
├── dependency_resolver.py # OCID resolution logic
└── resources/
    ├── base_resource.py # Abstract base class
  &   ├── vcn_resource.py # VCN implementation
  &   ├── subnet_resource.py # Subnet implementation
  &   └── instance_resource.py # Instance implementation

Factory Registration

Resources register themselves at import, enabling dynamic extension without modifying core logic.
