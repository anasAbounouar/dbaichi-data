What is Pulumi?

Core Concepts

  • Programs: Infrastructure defined as executable code in Python
  • Providers: Plugins interfacing with cloud APIs (OCI, AWS, Azure)
  • Stacks: Isolated deployments for different environments
  • State: Backend storage tracking infrastructure state and changes

Key Benefits

  • ✓ Use familiar programming languages (Python, TypeScript, Go)
  • ✓ Leverage existing tools and testing frameworks
  • ✓ Share and reuse components across projects
  • ✓ Strong typing and IDE support

Layered Architecture Design

Configuration Layer

Stack-specific JSON files define infrastructure requirements using human-readable resource names and direct parameter mapping to OCI services

Orchestration Layer

Python-based engine coordinates configuration loading, dependency resolution, resource factory operations through single-loop processing

Provisioning Layer

Pulumi SDK interfaces with OCI provider APIs to create, update, and manage infrastructure resources according to processed configuration

Integration Layer

GitLab CI/CD pipelines provide automated workflow management, including preview generation, approval gates, and deployment execution

Component Architecture

ConfigLoader

  • • Loads stack-specific JSON files
  • • Performs initial validation
  • • Maps stack names to config paths

ResourceFactory

  • • Dynamic registry: JSON → Python classes
  • • Automatic registration during import
  • • Consistent creation interface

DependencyResolver

  • • Converts logical names to OCIDs
  • • Handles internal and external references
  • • Ensures correct creation order

ExecutionEngine

  • • Orchestrates resource creation
  • • Coordinates all components
  • • Generates pipeline outputs

Factory Pattern Implementation

Key Benefits

✓ Dynamic Creation: No hardcoding of resource types
✓ Easy Extension: Add new resources without core changes
✓ Simplified Testing: Easy to test individual types

Dependency Resolution Strategy

Resolution Logic

  1. 1. Logical names in JSON (e.g., "production-vcn") are resolved to actual OCIDs at runtime
  2. 2. OCIDs starting with "ocid1." pass through directly for external resources
  3. 3. Internal resource registry maintained during execution
  4. 4. Missing dependencies trigger clear error messages with specific resource identification

Class Hierarchy & Inheritance Structure

Abstract Base Class Design

✓ Defines common resource creation interface (template method pattern)

✓ Provides shared functionality including resource tagging and validation

✓ Enforces consistent implementation patterns across all resource types

✓ Enables polymorphic resource handling within the factory system
